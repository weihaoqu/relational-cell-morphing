Cost Analysis Extensions with PickK + Prophecy

Five extensions of the PickK + prophecy technique to different algorithms, demonstrating its generality beyond the CountPositive baseline.

PREREQUISITE

The core PickK + prophecy technique is developed step-by-step in the CountPositive staged tutorial (Stages 1 → 2a → 2b). This page covers extensions beyond that baseline, plus async analysis and research findings.

What This Is

The core PickK + prophecy framework was developed for CountPositive: counting how many array elements are positive, then bounding how much the count can differ between two related arrays. These extensions show the technique applies broadly to different branch conditions, cost magnitudes, mutable arrays, value-dependent costs, and asymmetric stride patterns.

KEY IDEA

PickK dynamically re-focuses the distinguished cell to discover positions where A1[i] = A2[i].
Prophecy d0 promises at least d0 such equal positions exist.
At termination, the cost difference is bounded by a function of n - d0 (the unequal positions).

The Extensions

1. CountInRange — Different Branch Condition

CountInRange(A, n, lo, hi):
    cost := 0
    for i = 0 to n-1:
        if lo ≤ A[i] and A[i] ≤ hi:
            cost++

Bound: cost1 - cost2 ≤ n - d0

Insight: The specific branch condition (lo ≤ x ≤ hi vs x > 0) is irrelevant to the encoding. What matters is: equal values take the same branch (cost neutral), unequal values may diverge (worst case ±1). The PickK + prophecy structure is identical to CountPositive.

2. ExpensiveBranch (Nested) — Non-Unit Cost

ExpensiveBranch(A, n):
    cost := 0
    for i = 0 to n-1:
        if A[i] > 0:
            cost += 2          (* e.g., two array writes *)

Bound: |cost1 - cost2| ≤ 2(n - d0)

Insight: When the branch costs ±2 instead of ±1, each disagreeing position changes the cost difference by up to 2. The coefficient appears naturally in the bound. Equal positions (discovered by PickK) still contribute zero cost difference.

3. InplaceMap — Mutable Array

InplaceMap(A, n):
    for i = 0 to n-1:
        x = A[i]
        A[i] = f(x)           (* f(x) = x > 0 ? x+1 : 1 *)

Bound: cost1 - cost2 ≤ n - d0

Insight: Unlike read-only scans, InplaceMap writes back to the array. Cell morphing must track values that change during execution. The key property: f preserves equality (ak1 = ak2 implies f(ak1) = f(ak2)), so equal positions remain equal after mutation. This shows PickK + prophecy works with mutable arrays, not just scans.

4. ConditionalCost — Value-Dependent Cost with Epsilon

ConditionalCost(A, n):
    cost := 0
    for i = 0 to n-1:
        if A[i] > 0:
            cost += A[i]

Bound: cost1 - cost2 ≤ (n - d0) · ε

Insight: This is the key example that requires all three ingredients: PickK, prophecy, and an ε bound. The cost is value-dependent (cost += A[i], not cost++), so pure counting cannot bound it. At unequal positions, the Lipschitz property |max(a1,0) - max(a2,0)| ≤ |a1 - a2| ≤ ε gives a per-step bound of ε. At equal positions (d0 of them), the contribution is exactly zero.

5. Stride Analysis — Two Different Programs

P1(A1, n):                    P2(A2, n):
  for i1 = 0 to n-1:            for i2 = 0 to n-1 step 2:
    if A1[i1] > 0: cost1++        if A2[i2] > 0: cost2++

Bound: cost1 - cost2 ≤ n - d0

Insight: P1 visits every position (stride 1), P2 visits even positions only (stride 2). An epoch model absorbs the stride difference: each epoch j covers positions 2j and 2j+1, where P1 processes both and P2 processes only 2j. This gives a synchronized encoding with no scheduler. At HIT-equal epochs, the even position cancels and only the odd position can add ±1.

Every example returns UNSAT for its violation query, confirming the cost bound cannot be exceeded. The PickK + prophecy technique generalizes across branch conditions, cost magnitudes, mutable state, value-dependent costs, and asymmetric stride patterns.

The Three-Layer Architecture

Layer 1: Standard Cell Morphing (foundation)
  Track one distinguished cell (k, ak1, ak2, bk)
  HIT/MISS abstraction: i = k → values known; i ≠ k → unknown

Layer 2: PickK (existential morphing)            ← NEW
  Functional predicate PickK(state..., kp) re-chooses which cell to track
  Two phases: searching (move spotlight) and settled (keep current cell)

Layer 3: Prophecy Counting (d, d0)               ← NEW
  d0: "at least d0 positions have A1[i] = A2[i]"
  d: counter of discovered equal positions so far
  When d ≥ d0: prophecy fulfilled, stop searching

How They Work Together

PickK says: "focus on index i right now"
              ↓
Cell morphing says: "at this index, bk=1 means values are equal"
              ↓
Prophecy says: "that's another good position! d' = d + 1"
              ↓
Cost tracking says: "equal values → same branch → cost unchanged"
              ↓
At termination: "found d ≥ d0 good positions, so cost diff ≤ f(n - d0)"

Key Takeaways

  1. PickK is essentially free: Adding the functional predicate introduces no meaningful overhead to invariant synthesis. In some cases it is even faster than fixed-k encodings.
  2. Synchronized > Asynchronous for deterministic loops: For loops with fixed iteration order, sync encodings are ~4000x faster (seconds vs hours). Async is only needed when runs have data-dependent control flow.
  3. Clause count dominates variable count: Going from 5 to 9 variables barely matters. Going from 5 to 12 Horn clauses is catastrophic. Minimize clauses, not variables.
  4. The framework is modular: Each extension reuses the same Init/Transition/Wit/PickK/Goal structure. Only the per-step cost model changes.

Extension Deep Dives

Detailed walkthroughs of each extension encoding — transition snippets, key insights, and comparisons with the CountPositive baseline.

How to Read These Extensions

All extensions (and the CountPositive baseline) follow the same template. Only three things change:

What changesWhere in encodingExamples
Cost per stepHIT-unequal and MISS cases±1, ±2, ±ε
Cell mutationHIT-equal and HIT-unequal casescopy (read-only) vs f(x) (mutable)
Input precondition / boundInit clause and goalreplace only vs replace + L∞

Everything else — PickK rules, Wit, discovery counter d, prophecy d0, goal guard d ≥ d0 — is identical. The machinery is reusable; only the per-step cost model changes.

1. CountInRange — Branch Condition Abstraction

CountInRange(A, n, lo, hi):
    cost := 0
    for i = 0 to n-1:
        if lo ≤ A[i] and A[i] ≤ hi:
            cost++

Property: cost1 − cost2 ≤ n − d0

KEY INSIGHT

The branch condition (lo ≤ x ≤ hi vs x > 0) is irrelevant. The encoding only asks: are the values equal? Equal → same branch → cost neutral. Unequal → branches may differ → ±1. Any program of the form if P(A[i]): cost++ has the same encoding.

What the encoding sees at each step

Is i = kp? (HIT or MISS)
If HIT: is bkp = 1? (equal or unequal)

Equal → same branch (regardless of WHAT the branch IS) → c' = c
Unequal → branches may differ → c' = c + 1
MISS → unknown → c' = c + 1

Transition (identical to Stage 2a)

(
  (* HIT equal: same value => same branch => cost neutral *)
  (i = kp and bkp = 1
   and c' = c       and d' = d + 1)
  or
  (* HIT unequal: range check may differ, cost +1 *)
  (i = kp and bkp = 0
   and c' = c + 1   and d' = d)
  or
  (* MISS: unknown, worst case +1 *)
  (i <> kp
   and c' = c + 1   and d' = d)
)
Result: UNSAT (5.0s, 13 iterations) — bound n−d0. Same structure as CountPositive. The branch condition is irrelevant to the proof.

2. ExpensiveBranch — Non-Unit Cost

ExpensiveBranch(A, n):
    cost := 0
    for i = 0 to n-1:
        if A[i] > 0:
            cost += 2          (* expensive: e.g., two array writes *)

Property: |cost1 − cost2| ≤ 2·(n − d0)

KEY INSIGHT

The branch costs 2 instead of 1. Each disagreeing position changes the cost difference by up to 2. The coefficient flows through to the bound: 2·(n − d0).

What changes from CountPositive

CountPositive:      HIT unequal / MISS → c' = c + 1     (unit cost)
ExpensiveBranch:    HIT unequal / MISS → c' ∈ [c-2, c+2]  (cost ±2)

Transition (only cost magnitude differs)

(
  (* HIT equal: still free *)
  (i = kp and bkp = 1
   and c' = c                     and d' = d + 1)
  or
  (* HIT unequal: cost ±2, not ±1 *)
  (i = kp and bkp = 0
   and c' ≤ c + 2 and c' ≥ c - 2   and d' = d)
  or
  (* MISS: same ±2 *)
  (i <> kp
   and c' ≤ c + 2 and c' ≥ c - 2   and d' = d)
)

Goal

c > 2 * (n - d0) :-         (* coefficient 2 appears in bound *)
  Inv(...), n ≤ i, d ≥ d0.

PCSAT discovers the invariant c ≤ 2·(i − d). At termination: c ≤ 2·(n − d0).

Result: UNSAT (8.2s, 14 iterations) — bound 2(n−d0). Linear scaling of the PickK+prophecy bound.

3. InplaceMap — Mutable Arrays

InplaceMap(A, n):
    for i = 0 to n-1:
        x = A[i]               (* read *)
        A[i] = f(x)            (* write back: f(x) = x>0 ? x+1 : 1 *)

Property: |cost1 − cost2| ≤ n − d0 (cost counts iterations where the two runs take different branches of f)

KEY INSIGHT

Unlike read-only scans, InplaceMap writes to the array. Cell values mutate during execution. The critical requirement: f preserves equality — if ak1 = ak2, then f(ak1) = f(ak2). Equal cells stay equal after mutation, so PickK's discovery mechanism is undisturbed.

f preserves equality

ak1p = ak2p = 5  → f(5) = 6,  f(5) = 6  → ak1' = ak2' = 6  ✓
ak1p = ak2p = −3 → f(−3) = 1, f(−3) = 1 → ak1' = ak2' = 1  ✓
ak1p = ak2p = 0  → f(0) = 1,  f(0) = 1  → ak1' = ak2' = 1  ✓

Transition: Cell Mutation is Explicit

The HIT cases explicitly compute f — this is where InplaceMap differs from read-only examples:

(
  (* HIT equal: f preserves equality *)
  (i = kp and bkp = 1
   and (
     (ak1p > 0 and ak1' = ak1p + 1)      (* f(x) = x+1 when x>0 *)
     or
     (ak1p ≤ 0 and ak1' = 1)             (* f(x) = 1 when x≤0 *)
   )
   and ak2' = ak1'                          (* equal in → equal out *)
   and c' = c and d' = d + 1)
  or
  (* HIT unequal: apply f independently *)
  (i = kp and bkp = 0
   and (
     (ak1p > 0 and ak1' = ak1p + 1) or (ak1p ≤ 0 and ak1' = 1)
   )
   and (
     (ak2p > 0 and ak2' = ak2p + 1) or (ak2p ≤ 0 and ak2' = 1)
   )
   and c' = c + 1 and d' = d)
  or
  (* MISS: cell unchanged — not the position being mapped *)
  (i <> kp
   and ak1' = ak1p and ak2' = ak2p
   and c' = c + 1 and d' = d)
)

4. ConditionalCost — Value-Dependent (needs ε)

ConditionalCost(A, n):
    cost := 0
    for i = 0 to n-1:
        if A[i] > 0:
            cost += A[i]       (* cost is the VALUE itself *)

Property (mixed metric): d0 equal positions + all positions ε-close ⇒ cost1 − cost2 ≤ (n − d0) · ε

KEY INSIGHT

This is the only example requiring all three ingredients (PickK + prophecy + ε). Without ε, cost per step is unbounded. The Lipschitz property of max(x,0) converts input closeness into cost closeness: |max(a1,0) − max(a2,0)| ≤ |a1 − a2| ≤ ε.

Why ε is Required (but not in examples 1–3)

CountPositive:     cost += 1   → cost diff per step ∈ {0, ±1}  → bounded ✓
ExpensiveBranch:   cost += 2   → cost diff per step ∈ {0, ±2}  → bounded ✓
InplaceMap:        branch cost → cost diff per step ∈ {0, ±1}  → bounded ✓

ConditionalCost:   cost += A[i] → cost diff per step = ???    → UNBOUNDED!
  If a1[i]=100, a2[i]=0: cost diff is 100. No fixed bound.

The Three Ingredients

IngredientWhat it providesWithout it
PickKDiscovers equal positions dynamicallyBound is (n−1)·ε
Prophecy (d0)Parameterizes by input similarityMust commit to specific count
Epsilon (ε)Bounds per-step cost at unequal positionsCost difference unbounded
Result: UNSAT (11.5s, 15 iterations) — bilinear bound (n−d0)·ε. First example requiring all three ingredients. See CountPositive Stage 2b for the full walkthrough.

5. ArraySum Mixed Metric

The canonical value-dependent example. No branches, no conditionals — the cost difference comes purely from value differences. Combines replace metric (d0 equal positions) with L∞ metric (ε-closeness).

Precision Hierarchy

|sum1 − sum2| ≤
  n · max_val        (no info)
  n · ε               (pure L∞, standard robustness)
  (n − 1) · ε         (fixed-k + ε)
  (n − d0) · ε        (PickK + prophecy + ε) ← this encoding
  0                     (identical arrays, d0 = n)

Approach Comparison

No morphing (counting only)n · max_valNo value info at any position
Fixed-k + εn · εOne cell tracked, rest bounded by ε

Result: UNSAT (25s) — bound (n−d0)·ε. Tightest non-trivial bound. Encoding structurally identical to ConditionalCost.

6. Stride Analysis (Epoch Model)

Handles two different programs (stride-1 vs stride-2) by grouping steps into epochs instead of async scheduling.

P1(A1, n):                    P2(A2, n):
  for i1 = 0 to n-1:            for i2 = 0 to n-1 step 2:
    if A1[i1] > 0: cost1++        if A2[i2] > 0: cost2++

The Epoch Model

Each epoch covers 2 P1 positions and 1 P2 position. The epoch counter is synchronized — no scheduler needed. P1 processes both 2j and 2j+1, P2 processes only 2j.

Key Difference: HIT-equal Is Not Cost-Neutral

The odd position within the epoch is invisible to P2 but visible to P1. HIT-equal cost can increase by 0 or 1. PCSAT discovers the invariant: c ≤ 2j − d.

Result: UNSAT — 9 variables, 5 clauses. Same encoding size as CountPositive. Stride asymmetry adds no complexity.

The Async Frontier

When synchronized models don't apply, what happens? Deep analysis of the async bottleneck and strategies for tractability.

Why Async CountPositive Takes 3 Hours

The smoking gun: CountPositive encoded both ways — sync (2.5s) vs async (~3 hours). 4000× speedup just from modeling choice.

Why More Clauses = Exponentially Harder

PCSAT must find an invariant that is simultaneously inductive under ALL clauses. Each clause imposes constraints on the invariant shape. With 5 clauses, the constraint system is tractable. With 11 clauses, the interaction between constraints causes combinatorial explosion.

When Is Async Genuinely Needed?

CountPositive has deterministic iteration order — both runs visit 0, 1, 2, …, n−1 in order. The async scheduler adds spurious interleavings. But some algorithms truly require it:

PatternWhy runs divergeExample
Data-dependent strideStep size varies by valuei += A[i]>0 ? 2 : 1
Data-dependent skipSome iterations skippedKruskal: add/skip edge
Early terminationLoop exits at different pointsLinear search, find-first
Different programsStructurally different loopsStride-1 vs stride-2
Decision Rule

If BOTH runs iterate for i = 0 to n-1 (same fixed order) → Use SYNC. Single counter. ~5 clauses.

If runs CAN visit different indices → Use ASYNC. Two counters. ~11 clauses.
→ Consider EPOCH MODEL if stride ratio is fixed (e.g., stride-1 vs stride-2).

Case Study: StrideProcess

A genuinely async algorithm where the stride depends on data:

StrideProcess(A, n):
    cost := 0
    i := 0
    while i < n:
        if A[i] > 0:
            cost++
            i += 2             (* skip next *)
        else:
            i += 1             (* normal step *)

When A1 and A2 differ, the runs take different strides, so the loop counters diverge:

A1 = [3, -1, 5, 2, -4]    A2 = [3, -1, -2, 2, -4]
                                    ^-- differs at index 2

Run 1: i=0 (A1[0]=3>0, skip) → i=2 (A1[2]=5>0, skip) → i=4 (A1[4]=-4, step) → i=5. Done.
Run 2: i=0 (A2[0]=3>0, skip) → i=2 (A2[2]=-2, step) → i=3 (A2[3]=2>0, skip) → i=5. Done.

After step 2: Run 1 at i=4, Run 2 at i=3 — DIVERGED!

No sync model can capture this — the scheduler must interleave the runs to explore all possible divergence patterns.

The PickK Dilemma in Async

With two counters i1 ≠ i2, which index should PickK focus on?

OptionFocus onImplication
A: Always i1Run 1’s positionNo info at Run 2’s position
B: Always i2Run 2’s positionNo info at Run 1’s position
C: Only when i1=i2Aligned positionsBoth runs see the same element

Option C is most promising — when counters align, both runs see the same position, and PickK can discover equality. But the scheduler must arrange alignment, which requires coordination.

The epoch sync model absorbs the stride difference: each epoch j covers positions 2j and 2j+1 for P1 and position 2j for P2. No scheduler needed. Already verified UNSAT.

The full async model is the real test: can PCSAT handle full scheduler + PickK + prophecy on a genuinely async problem? Here, unlike CountPositive, the scheduler isn't adding spurious interleavings — it's actually needed to arrange alignment.

The Scheduler's Meaningful Role

TT (both step): i1→i1+1, i2→i2+2. At alignment (i1=i2=2j), PickK discovers equality (d++). Now i1=2j+1, i2=2j+2 — misaligned.
TF (P1 catches up): i1→i1+1. Now i1=2j+2, i2=2j+2 — realigned!

Pattern: TT, TF, TT, TF, … gives epochs where PickK can focus on each even position. The scheduler arranges alignment; PickK exploits it.

Strategies for Making Async Tractable

Four approaches under investigation:

1. Simplify the Scheduler
Restrict interleavings to the TT-TF pattern instead of allowing arbitrary scheduling. Fewer clauses, more structured search.
2. Two-Counter Sync (Most Promising)
Both runs step every iteration but advance by different amounts. No scheduler needed — just two counters that increment differently. Gives sync-like clause count (~5) with async-like expressiveness.
3. Prophecy Over Alignment
Prophesy the alignment pattern itself: “the runs will be aligned at least d0 times.” Moves alignment reasoning from the scheduler into the prophecy framework.
4. Gap Bounds Instead of Exact Tracking
Instead of tracking exact counter positions, track the gap |i1 − i2| and bound how it evolves. Reduces state without losing the essential cost information.
Bottom Line

Sync works great for deterministic-order loops (>90% of examples). Async + PickK for execution cost is an open problem — the scheduler's 6 extra clauses cause combinatorial explosion. The two-counter sync idea is the most promising path forward: both runs step together, different amounts, no scheduler overhead.

Research Summary

Key findings, the full results table, cost pattern taxonomy, and an encoding checklist for practitioners.

Problem and Contributions

Standard cell morphing tracks one fixed cell k. For output properties (robustness, sensitivity), one cell suffices because ε-closeness absorbs the unknown cells. For cost properties, each cell's contribution matters — one cell gives at best an n − 1 bound.

PickK + Prophecy breaks this ceiling:

  • PickK: Dynamically re-focuses the distinguished cell to discover positions where A1[i] = A2[i]
  • Prophecy d0: “At least d0 positions are equal” — parameterizes the bound by input similarity
  • Result: bound f(n − d0) instead of f(n − 1)
See the CountPositive staged tutorial for the full step-by-step development: Stage 1 (fixed-k ceiling) → Stage 2a (PickK + prophecy) → Stage 2b (+ epsilon).

Key Findings

1. PickK Has Zero Overhead

Sync+PickK (9 vars, 5 clauses) was faster than pure counting (5 vars, 3 clauses): 2.5s vs 5s. The functional predicate adds no meaningful overhead. PCSAT handles existential morphing naturally.

2. PickK Strictly Improves Over Fixed-k

Fixed-k proves n − 1 (and cannot prove n − 2). PickK proves n − d0 for arbitrary d0 ≤ n. This is a strict improvement that degrades gracefully from n (d0=0, no info) to 0 (d0=n, identical arrays).

3. Async Scheduler Is the Entire Bottleneck

Sync to async: 2.5s → ~3 hours (4000× slower). The scheduler adds 6 Horn clauses. For cost encodings with PickK, this is the dominant performance factor. Don’t use async for deterministic-order loops.

4. Extra Clauses Hurt (Replicated Finding)

Adding a “sound but unnecessary” PickK fallback rule made things slower. This echoes the Dijkstra Approach A failure (15+ hours with one extra TT clause). Now replicated across two different encoding families.

Rule of thumb: Only add a Horn clause if it carries information the solver needs. Don’t constrain cases that the solver handles via existential freedom.

5. PickK Is Genuinely Necessary for Value-Dependent Cost

CountPositive can be proved by pure counting (±1 per step). ConditionalCost and ArraySum cannot — cost is A[i], which is unbounded without cell morphing. PickK + ε gives the per-step bound. Counting has no access to values.

Open Questions

  1. Multiple distinguished cells: Could PickK1 + PickK2 track two cells simultaneously? Would add variables but might give tighter bounds for algorithms with two interacting cost-relevant positions.
  2. Tighter bounds at HIT-unequal: For ArraySum, the HIT-unequal case could split further (both positive → tighter Lipschitz bound). Can PCSAT handle the extra cases without blowup?
  3. Non-linear cost functions: All examples have linear cost (f(n−d0) where f is linear). What about cost += A[i]²? The Lipschitz constant grows, making the per-step bound less useful.
  4. Scalability to larger programs: The framework handles single-loop programs. Multi-loop or nested-loop algorithms would need multiple PickK phases or sequential cell morphing.
  5. Connection to quantitative information flow: Cost sensitivity ↔ information leakage? The d0 parameter ↔ a measure of “how much the adversary knows.”

Required Horn Clauses (Sync)

1. Init:       Inv(0, n, k, ak1, ak2, bk, 0, d0, 0) :- precondition.
2. Transition:  Inv(primed) :- Inv(unprimed), PickK(...,kp), Wit(...),
                  (HIT-equal or HIT-unequal or MISS), i' = i+1.
3. Witness:     Wit(state) :- Inv(state), i < n.
4. PickK search: PickK(state, i) :- Inv(state), i < n, d < d0.
5. PickK settle: PickK(state, k) :- Inv(state), d ≥ d0.
